# Feature: BetterBond Commission Payments POC

## Summary

BetterBond's existing commission payments process is manual and error-prone. This POC gives internal payments administrators a single interface to track, park, batch-process, and invoice agent commissions — demonstrating what a full rewrite could deliver. Real estate agency staff (viewers) can access dashboard summaries and historical payment batches in a read-only capacity.

The feature ships as a Next.js 16 app with NextAuth credentials authentication, role-based access (admin / viewer), a Dashboard with charts and agency summary, a Payment Management screen (admin-only) for park/unpark/bulk operations and batch initiation, a Payments Made screen for historical batch browsing and invoice downloads, and a Users screen (admin-only) for provisioning accounts. The UI uses the MortgageMax brand palette with both light and dark themes, South African currency formatting, and POPIA-compliant data minimisation.

2. **Epic 2: Dashboard** - The at-a-glance view of payment activity across agencies, with charts, summary tiles, aging report, and the agency summary grid that hands off into Payment Management. | Status: Pending | Dir: `epic-2-dashboard/`

5. **Epic 5: User Management (Admin only)** - Admin-only Users screen for provisioning accounts and assigning the admin or viewer role. | Status: Pending | Dir: `epic-5-user-management/`

## Epic Dependencies

- Epic 1: Auth, Shell, and Navigation Foundation (no dependencies — must be first). Delivers the NextAuth session, top nav, role guards, toast infrastructure, and theme switcher that every other epic relies on.
- Epic 2: Dashboard (depends on Epic 1). Needs the authenticated session, top nav, toast infrastructure, and role-aware access control in place. Independent of Epics 3–5 but sequenced before Epic 3 so the agency-row click-through into Payment Management is testable.
- Epic 3: Payment Management (depends on Epic 1; sequenced after Epic 2). Uses the session's user identity for `LastChangedUser`, the admin-only route guard, the toast infrastructure, and the agency hand-off from the Dashboard. Cannot parallel with Epic 2 because the dashboard-to-payment-management workflow needs both screens working end-to-end.
- Epic 4: Payments Made (depends on Epic 1; recommended after Epic 3). Technically only needs Epic 1, but is most valuable after Epic 3 so the full "initiate payment → appears in Payments Made" loop can be verified manually. Could parallel with Epic 3 if resourcing allowed.
- Epic 5: User Management (depends on Epic 1). Independent of Epics 2–4 — can parallel with any of them. Placed last to keep focus on the core commission-payment happy path first.

## Epic Ordering Rationale

Epic 1 ships the authenticated, role-aware shell first because everything else needs a session, a nav bar to hang off, and the toast/guard infrastructure. Epic 2 follows to deliver the first visible screen users see after sign-in. Epic 3 comes next so the Dashboard-to-Payment-Management hand-off can be exercised end-to-end. Epic 4 comes after Epic 3 so freshly created batches show up in the Payments Made history (making the whole payment lifecycle testable). Epic 5 is last because user provisioning, while required, is an admin-only side road that doesn't block the core POC demo.
